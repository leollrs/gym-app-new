## 🔁 Read this first: five systemic patterns

Most of the 99 findings are not independent bugs — they're instances of five root patterns. Fixing the pattern is far cheaper than fixing symptoms one by one, and it stops the next instance from being written.

### Pattern 1 — The 1000-row PostgREST cap (16 instances)
Supabase silently caps any `.select()` without an explicit `.range()`/pagination at 1000 rows (some code sets 500 or 2000). Every count, total, export, and campaign recipient list built by materializing rows and taking `.length` is **wrong once the table crosses the cap** — and it fails *silently*, looking complete. Most instances are P2 today only because no gym has 1000+ of anything yet; three are already P1 because the cap is 500 (segments) or the data grows fast (DMs, revenue).

**Fix once, systematically:** for counts use `select('*', { count: 'exact', head: true })` (no rows transferred); for recipient lists / exports, paginate explicitly. The worst offenders — anything that sends money, SMS, or push, or reports revenue — should be fixed now even though the data is small, because they're the ones that hurt when they silently drop people: `segmentFilters.js:44`, `notifications.js:36` (announcements/NPS), `AdminRevenue.jsx:159`, `SupportConsole.jsx:561` (DSAR). The rest can wait until you have a gym near the cap.

### Pattern 2 — The referral/points economy (13 instances, the money core)
The single most concentrated risk area. Points convert to real merch and free months, so every one of these is a real value leak. They cluster because the economy RPCs have been rewritten several times (0117 → 0372 → 0470 → 0480 → 0601) and each rewrite dropped a guard the previous one added: idempotency, per-gym monthly cap, same-gym check, row locking, admin-approval gating. **This whole subsystem deserves one focused server-side pass** that re-establishes, in one migration: (a) an idempotency/dedup key on every award, (b) `FOR UPDATE` row locks on balance-debit paths, (c) the per-gym monthly referral cap, (d) same-gym enforcement, and (e) making the admin "Approve" button call `safe_complete_referral()` instead of a bare status flip. All server-fixable.

### Pattern 3 — RLS policies missing `WITH CHECK` / a role gate (8 instances, includes the only P0)
Several `FOR ALL` / INSERT / UPDATE policies pin the *owner* (`profile_id = auth.uid()`) but leave `gym_id` and other columns unconstrained, or gate on same-gym without checking the caller's *role*. The P0 (`trainer_clients` self-assign, `0291`) is the severe end; `check_ins` (`0002:371`), `workout_sessions`/`personal_records`/`member_goals` (`0002:293`), and `admin_gift_reward` (`0471`) are the same shape. **Fix pattern:** add `WITH CHECK` clauses that pin `gym_id` to the caller's own gym and, where relevant, assert the caller's role inside `SECURITY DEFINER` bodies rather than trusting the grant. All server-fixable.

### Pattern 4 — Non-idempotent writes that double-fire on retry (6 instances)
On flaky gym WiFi, the dangerous case is "the write committed but the response was lost" → the client retries → a second row / a second award. Hits workout completion (duplicate session + double XP/PR points), win-back reward gifting, and the referral paths. **Fix pattern:** a stable client-generated key per logical action + upsert/short-circuit on it (the QR check-in already does this correctly — copy that shape).

### Pattern 5 — `${cssVar}` + 2-hex alpha concatenation → invalid transparent CSS (7 instances)
Code builds a faux-transparent color as `` `${someColor}18` ``, but `someColor` is a CSS `var(--x)`, so the result is `var(--x)18` — invalid, so the background/border renders transparent. Cosmetic (P2) but easy to sweep: replace with `color-mix(in srgb, var(--x) 18%, transparent)`. Instances in Nutrition, PersonalRecords, CheckIn, AdminSegments, AdminChurn, ClassDetailModal.

## 🔴 The P0 — fix first (server-side, ships today)

#### 🔴 P0 · 🟢 server (ship today) — Any member can self-assign as a trainer to any same-gym member (trainer_clients INSERT has no role gate) → full read/write breach of that member's coaching, health, PR, goal & nutrition data
`supabase/migrations/0291_fix_trainer_rls_gym_id.sql:25`

The entire trainer authorization surface trusts one thing: an active row in `trainer_clients` for `auth.uid()`. Both gate helpers — `public._can_manage_client(client_id)` (0463_multi_role_sweep_admin_trainer.sql:88-91) and `public.is_trainer_of(client_id)` (0211_security_fix_trainer_policies.sql:14-22) — return TRUE whenever an active `trainer_clients` row exists with `trainer_id = auth.uid()` and matching gym, and NEITHER checks that the caller actually holds the trainer role. The `trainer_clients` INSERT policy `trainer_clients_insert_trainer` (current definition, 0291:25-29) is `WITH CHECK (trainer_id = auth.uid() AND gym_id = public.current_gym_id())` — no role check, no member consent. Table-level INSERT is reachable by every authenticated user: migration 0476_relock_materialized_views_grants.sql:14-16 documents the live blanket `GRANT ALL ON ALL TABLES ... TO anon, authenticated` drift ("same disease as profile_lookup, see 0473"), and `trainer_clients` was never included in any corrective REVOKE (unlike profile_lookup/0473, reward tables/0429, matviews/0476, qr nonces/0478, direct_messages/0536). `is_active` defaults TRUE (0001_initial_schema.sql:163) and `current_gym_id()` reads the caller's own profiles.gym_id (tamper-proof — gym_id changes are blocked by guard_profile_update, 0027). Net: a plain member escalates to "trainer of" any other member in their gym with a single write. That immediately unlocks the trainer-read/write RLS on the victim's checkin_responses/habits/habit_logs (0500), body_measurements + body_weight_logs (0029/0572), personal_records + member_goals + gym_program_enrollments DELETE (0527), member_onboarding UPDATE incl. allergies/diet (0575), the victim's private social feed via get_friend_feed's trainer arm (0527), plus every `_can_manage_client`-gated RPC (set_client_fee, mark_client_paid, trainer_send_cue, set_client_schedule). The app's own Add-Client flow is a bare client-side upsert (src/pages/trainer/TrainerClients.jsx:318-323), so the exploit is that exact PostgREST request with a substituted victim client_id. Gym boundary DOES hold (intra-gym only); cross-gym is blocked. Victim gets a 'new_trainer' push (0530) but only AFTER the row — and thus the access — exists.

**Repro:** As an ordinary member (role='member') of gym G, using your own valid Supabase JWT, POST to the REST endpoint for trainer_clients (identical to TrainerClients.jsx handleAdd): upsert {trainer_id: <your own uid>, client_id: <any other member of G>, gym_id: <your gym_id>, is_active: true}. The insert passes RLS. Then SELECT from body_measurements / personal_records / checkin_responses / member_goals filtered to that client_id — rows are now returned (previously blocked). You can also INSERT habits / checkin_assignments for the victim, DELETE their gym_program_enrollments, and call set_client_fee/trainer_send_cue against them.

**Fix:** Add a caller-role gate to the trainer_clients INSERT policy WITH CHECK: require the inserter to actually be staff, e.g. `trainer_id = auth.uid() AND gym_id = public.current_gym_id() AND public.is_staff()` (or an EXISTS on profiles/profile_lookup requiring role/additional_roles trainer|admin). Defense-in-depth: also add the same role assertion inside `_can_manage_client` and `is_trainer_of` (caller must hold trainer/admin), and issue `REVOKE INSERT, UPDATE, DELETE ON public.trainer_clients FROM anon, authenticated;` to kill the blanket-grant residue (admin/trainer assignment already flows through role-checked paths + RLS, and legit trainer self-add would go through the role-gated policy). Ships as a SQL migration — no app update required. Audit every other base table for the same blanket-grant residue.

#### 🟠 P1 · 🟡 either — Finish-workout Retry after a committed-but-lost RPC response creates a duplicate session + double points/XP/streak _(found 2×)_
`src/pages/ActiveSession.jsx:2863`

handleFinish() calls the complete_workout_v2 RPC (which wraps complete_workout / migration 0483). complete_workout does an UNCONDITIONAL `INSERT INTO workout_sessions ...` (supabase/migrations/0483_economy_hardening.sql:244) with no idempotency key and no dedup on (profile_id, started_at) — there is no unique constraint on workout_sessions to make a re-insert fail (verified: only constraint is the rating CHECK in 0561). The client's only duplicate guard is `if (saving) return` (ActiveSession.jsx:2809), which only blocks a *concurrent* in-flight call. On spotty gym WiFi (the actual deployment environment), the dangerous window is: the RPC commits server-side, then the HTTP response is dropped before the client receives it. The client hits the catch block, keeps the localStorage draft, sets `setSaving(false)` (line 2967), and shows a prominent 10-minute 'Couldn't save your workout. Tap retry.' toast whose action re-invokes `handleFinishRef.current?.()` (line 2977). Because `started_at` is stable but `completed_at` is regenerated each call and the RPC never checks for an already-recorded session, the retry inserts a SECOND workout_sessions row. Points/XP are keyed by dedup_key `workout:<session_id>` — but the session_id is freshly generated on the retry, so the key differs and points ARE re-awarded (+50 XP workout, +100/PR, streak_7/30, first_weekly). Result: duplicate workout in the log, a duplicate 'workout_completed'/'pr_hit' post in the social feed, and double reward points (which redeem for real merch / free months → money/points leak).

**Repro:** 1) Start and complete a workout. 2) Simulate the gym-WiFi failure mode: let complete_workout commit on the server but drop the response (e.g., kill connectivity right as Finish is tapped, after the DB write lands). 3) The retry toast appears; tap Retry once connectivity returns. 4) Observe two workout_sessions rows for the same session, two feed posts, and points awarded twice.

**Fix:** Make completion idempotent: have the client generate a stable client_session_id (UUID) once per session (persist it in the draft) and pass it into complete_workout; upsert on a UNIQUE(profile_id, client_session_id) and short-circuit (return the existing session_id) if the row already exists. Alternatively add UNIQUE(profile_id, started_at) on workout_sessions so the retry insert fails cleanly, and have complete_workout catch 23505 by fetching+returning the existing session instead of raising.

#### 🟠 P1 · 🟢 server (ship today) — pr_hit points (100) + public "New PR" feed item are awarded without re-checking the PR actually beat the stored record
`supabase/migrations/0483_economy_hardening.sql:516`

In the final live complete_workout (0483), the personal_records upsert correctly gates the write on the new estimate beating the old one: `ON CONFLICT ... DO UPDATE SET ... WHERE EXCLUDED.estimated_1rm > personal_records.estimated_1rm` (line 340). But the two downstream loops that (a) post the 'pr_hit' activity_feed_items (lines 368-398) and (b) award C_PR_XP = 100 points via add_reward_points (lines 515-538) do NOT re-apply that comparison. #### 🟠 P1 · 🟡 either — Win-back reward gift is non-idempotent; a send failure + retry double-issues claimable rewards
`src/pages/admin/components/WinBackModal.jsx:74`
