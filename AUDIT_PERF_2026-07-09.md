## 🔁 Read this first — the patterns

### Pattern 1 — the `max_rows = 1000` cap (47 findings, the dominant theme)

Supabase clamps **every** response to 1000 rows (`supabase/config.toml` `max_rows = 1000`). Two things make this a landmine here:

1. **The `.limit(10000)` / `.limit(2000)` / `.limit(5000)` scattered across exports and analytics are _false safeguards_.** They read as "we support 10K rows," but `max_rows` clamps them to 1000 anyway. Raising the number does nothing — you must paginate. Anyone auditing the code (including us, first pass) reads them as safe; they aren't.
2. **supabase-js never throws on truncation.** The query returns 1000 rows and looks complete, so every downstream count/chart/export silently shows a partial or wrong number with zero signal.

Severity splits by _where_ it lands — this is the key call:
- **Member side → P2 (latent):** needs one member with 1000+ sessions/messages/PRs (years). Fine for a long time.
- **Admin/Platform analytics & exports → P1 (wrong data, near-term):** retention/cohort/attendance/revenue/NPS/Overview KPIs computed on a 1000-capped read are **understated and non-deterministic**, and a 40-member gym crosses 1000 check-in rows in ~25 days. Campaign/segment sends and CSV exports silently **drop recipients/rows**.

**Fix pattern (copy what's already in the repo):** `selectAllRows()` (`src/lib/churn/batchedSelect.js`, already used by `outreachAudience.js`) to page the full set for exports/campaigns; `{count:'exact',head:true}` (`overviewQuery.js:79`) for true totals + a "showing N of M" truncation warning; or a server-side aggregate RPC so the math never leaves Postgres. `GymsOverview` already uses `selectAllRows` correctly — it's the model.

### Pattern 2 — Functional-failure: things that are broken *now* (21 findings, scale-independent)
These don't wait for scale — they fail on the first user. This is what folding the codes/links lens in bought us; a pure perf pass misses all of it.

#### 🔴 P0 · 🟡 either · _audit-1_ — Entire invite pipeline hardcodes the dead `tugympr.app` domain AND send-invite's validator REQUIRES it (rejects the correct app.tugympr.com)
`supabase/functions/send-invite/index.ts:173`

The canonical web host is `app.tugympr.com` (appUrls.js:5, and the ONLY host in the iOS AASA / Android intent filters). But the member-invite flow is standardized on the dead `tugympr.app` domain in three client generators — AdminMembers.jsx:1058/1061 (invite QR value + printed label), InviteModal.jsx:40,107 (`https://tugympr.app/invite/${code}`), CreateInviteModal.jsx:147 — AND the send-invite edge function's guard at line 173 `if (inviteUrl && !/^https:\/\/([a-z0-9-]+\.)?tugympr\.app\//i.test(inviteUrl))` REQUIRES the URL to be a `tugympr.app` link, returning 400 `inviteUrl must be a tugympr.app https link` for anything else. This is a deadlock: (a) the invite email/SMS/QR all point recipients at `tugympr.app`, which is not the app's associated domain and (per the app's own config) resolves to nothing usable — the App Link never opens the app on either platform AND there is no web app served there, so the invitee hits a dead page; (b) the moment anyone fixes the client to emit the correct `app.tugympr.com/invite/...`, send-invite will REJECT it with a 400, silently failing the branded email/SMS send. The whole new-member acquisition path (the core B2B onboarding funnel) is broken end-to-end.

**Fix:** Single source of truth already exists: import `PROD_WEB_URL` from src/lib/appUrls.js (as Referrals.jsx:15 / Classes.jsx:13 / Challenges.jsx:8 already do) and build `${PROD_WEB_URL}/invite/${code}` in all three modals + AdminMembers QR. In send-invite/index.ts:173 change the regex to accept the canonical host, e.g. `/^https:\/\/([a-z0-9-]+\.)?tugympr\.com\//i` (matches app.tugympr.com). The edge-fn change ships instantly (server); the three client URL literals are client=app_update. `/invite/*` is already in the AASA + intent filter, so once assetlinks is fixed these open the app.

#### 🟠 P1 · 🟢 server · _audit-1_ — Win-back reward-QR (reward-qr edge fn) joins signature with COLON, but the scanner only accepts PIPE — every emailed/SMS'd/pushed win-back reward QR is rejected as "Invalid QR — please refresh in the app"
`supabase/functions/reward-qr/index.ts:179`

CONFIRMS the Phase A reward-qr finding and pins the exact break + the concrete delivery site. reward-qr builds the QR value as `qrValue = timestamped + ':' + sig` (index.ts:177-179), i.e. `gym-reward:{gymId}:{profileId}:{redemptionId}:{ts}:{sig}` — signature joined with a COLON and NO pipe anywhere. The scanner's contract for signed payloads is `payload:ts|signature` (PIPE): qrSecurity.js:21 and useSignedQR build in-app QRs as `${data.payload}|${data.signature}`, and scanRouter.handleScannedValue keys the verify branch on `trimmed.lastIndexOf('|')` (scanRouter.js:67). For a reward-qr string there is no '|', so lastPipe === -1, the verify branch is skipped, and scanRouter.js:94 (`trimmed.startsWith('gym-reward:') && !trimmed.includes('|')`) fires -> setError('Invalid QR — please refresh in the app') and returns null. The recipient CANNOT 'refresh in the app': this QR is delivered as a static PNG via WinBackModal.jsx:96 (`/functions/v1/reward-qr?id=${redemptionId}&format=png`) over SMS (mediaUrl, WinBackModal.jsx:148) and referenced in the email/push body ('Show the QR code above to staff'). The gifted redemption IS otherwise valid and claimable (admin_gift_reward inserts status='pending', points_spent=0, 0288:42-44; claim_redemption would succeed, 0311:28-92) — but the delivered artifact never reaches handleRewardRedemptionScan, so the claim side-effect never fires for the delivered QR. NOTE: the SAME redemption rendered in-app on the Rewards page works, because RedemptionQRModal (Rewards.jsx:265-272) builds `gym-reward:{gymId}:{userId}:{redemptionId}` and signs it through useSignedQR (pipe). So the break is isolated to the reward-qr edge-fn delivery leg. IMPORTANT for the sibling question: check-in QRs do NOT share this bug — check-in is signed exclusively via useSignedQR (pipe) and wallet check-in carries a bare qr_code_payload; the colon-vs-pipe mismatch exists ONLY in reward-qr.

**Fix:** Match the in-app signer contract in qrSecurity.js:21 (`${payload}|${signature}`); reward-qr is the only signed-QR producer that diverges from it.

#### 🟠 P1 · 🔵 app · _audit-1_ — applySegmentFilters caps base roster at .limit(500) — segment campaign SEND, recipient COUNT, CSV export, and 'Message All' DM all silently truncate to 500 members
`src/lib/admin/segmentFilters.js:44`

applySegmentFilters() resolves a member segment by fetching profiles with `.order('full_name').limit(500)` (line 44). #### 🟠 P1 · 🟡 either · _audit-1_ — FleetCostPanel MRR/margin contradicts the page's own bracket-priced MRR card (platform_cost_summary sums NULL monthly_price)

#### 🟠 P1 · 🟡 either — Bulk gym Deactivate/Activate bypasses pause_gym/unpause_gym — corrupts member membership_status across paths (banned members resurrected as active; or whole gym permanently locked out)
`src/pages/platform/GymsOverview.jsx:265`

The bulk fleet actions do a raw `.update({ is_active })` directly on `gyms` (bulkSetActive → GymsOverview.jsx:265; bulkSetPlan:273; bulkSetFeature:279 via applyBulk:246-262). They do NOT call the pause_gym / unpause_gym RPCs. The single-gym path in GymDetail was deliberately fixed to route through these RPCs — the code comment at GymDetail.jsx:692-694 states verbatim: 'route through pause_gym (0294) so each member's current status is snapshotted into pre_pause_status BEFORE everyone is deactivated (the old direct update made a later reactivate resurrect banned members as active)'. The bulk path reintroduces exactly that bug, and worse, creates a cross-path asymmetry: (A) Bulk Deactivate (is_active=false only): members' membership_status is never snapshotted to pre_pause_status and never set to 'deactivated'. If the operator later reactivates that gym via the single-gym Reactivate button (GymDetail.jsx:691 → unpause_gym, 0441), unpause_gym runs `SET membership_status = COALESCE(pre_pause_status,'active')` for ALL members. Since pre_pause_status is NULL (bulk never wrote it), EVERY member — including any who were legitimately 'banned', 'frozen', or 'cancelled' before the bulk deactivate — is forced to 'active'. Banned members silently regain full app access. (B) Bulk Activate (is_active=true only) on a gym that was Deactivated via the single-gym path (pause_gym set everyone to 'deactivated' and snapshotted pre_pause_status): bulk Activate flips is_active=true but never calls unpause_gym, so every member stays membership_status='deactivated'. AuthContext.jsx:308 blocks members whose blockedStatus is 'deactivated', so the gym shows as live/active in the fleet list while EVERY member is permanently locked out on the 'membership deactivated' screen, with pre_pause_status still holding their real status but never consulted and no UI to fix it. Both modes are silent — the operator sees a green success toast ('N gyms activated/deactivated', GymsOverview.jsx:257,268-269) and a healthy-looking gym.

**Fix:** Make the bulk activate/deactivate execs call the same RPCs the single-gym path uses, per selected gym: bulkSetActive(false) → loop `supabase.rpc('pause_gym',{p_gym_id:id})` (and set plan_type/subscription_tier='cancelled' to match GymDetail.jsx:711); bulkSetActive(true) → loop `supabase.rpc('unpause_gym',{p_gym_id:id})`. Better: add a SECURITY DEFINER `super_admin_bulk_pause_gyms(uuid[])` / `..._unpause_gyms(uuid[])` that does the snapshot/restore set-based in one statement, mirroring pause_gym/unpause_gym semantics, and report per-gym failures.

## Note on Phase E (codes/links end-to-end)
